# fly/ — deploy this branch to Fly.io with the sample sign-in providers

A [Fly.io](https://fly.io/) deployment of **this checkout** of
datasette-accounts (wheel built locally, not PyPI) plus the three sample
sign-in providers — `samples/{discord,github,bluesky}-auth` — loaded as
`--plugins-dir` modules. Modeled on `~/work/simonw/accounts-example-fly`,
which deploys the released PyPI packages instead.

This directory is git-ignored (`.git/info/exclude`) — it's a local deploy
harness, not part of the plugin.

- **App:** `accounts-providers-fly` (from `fly.toml`; scripts read it there)
- **URL:** https://accounts-providers-fly.fly.dev/

## Layout

| Path | Purpose |
| ---- | ------- |
| `pyproject.toml` / `uv.lock` | Deps: datasette (prerelease), the **local wheel** via a `[tool.uv.sources]` path, authlib (bluesky's DPoP JWTs), datasette-paper. |
| `wheels/`, `plugins/` | Generated by `scripts/sync.sh` — gitignored build artifacts from the parent checkout. |
| `run.sh` | Entrypoint: `datasette --internal /mnt/internal.db /mnt/data.db --create --plugins-dir plugins`. The volume-backed `--internal` is what makes accounts survive restarts. |
| `Dockerfile` / `fly.toml` | Same shape as accounts-example-fly: uv base image, `/mnt` volume, single machine. |
| `.env` (from `.env.example`) | Provider credentials + `DATASETTE_SECRET`, pushed as Fly secrets by `scripts/secrets.sh`. |

## Scripts (run in order the first time)

```sh
scripts/provision.sh     # 1. fly apps create + volume — NO deploy; prints the URL
                         # 2. register OAuth apps using that URL (below), fill .env
scripts/secrets.sh       # 3. push .env → fly secrets (--stage); generates DATASETTE_SECRET
scripts/deploy.sh        # 4. sync.sh + local Docker build + fly deploy --ha=false
scripts/bootstrap-db.sh  # 5. build internal.db locally, sftp it to /mnt, restart
```

`scripts/sync.sh` (run automatically by deploy.sh) rebuilds the frontend,
builds the wheel out of the parent checkout into `wheels/`, flat-copies the
three sample `*_auth.py` files into `plugins/` (flat, so no loader shim is
needed — one `--plugins-dir` sees them all), and re-locks (`--refresh-package
datasette-accounts`, because uv pins the wheel's sha256 and a rebuilt wheel
otherwise trips a hash mismatch).

## Why provision before deploying

The OAuth callback URLs embed the app's public URL, so the app must exist
(name reserved → URL known) before you can register the OAuth apps or set
`DATASETTE_BLUESKY_PUBLIC_URL`:

- **GitHub** — OAuth App with callback `https://<app>.fly.dev/-/github-auth/callback`
  → `DATASETTE_GITHUB_CLIENT_ID` / `DATASETTE_GITHUB_CLIENT_SECRET`
- **Discord** — application with redirect `https://<app>.fly.dev/-/discord-auth/callback`
  → `DATASETTE_DISCORD_CLIENT_ID` / `DATASETTE_DISCORD_CLIENT_SECRET`
- **Bluesky** — nothing to register (atproto public client: the instance
  serves its own client-metadata at `/-/bluesky-auth/client-metadata.json`)
  → just `DATASETTE_BLUESKY_PUBLIC_URL=https://<app>.fly.dev`

Any provider left blank in `.env` is inert (no login button, `start` 503s),
so partial configuration is fine.

## Bootstrapping the first admin (`scripts/bootstrap-db.sh`)

A fresh install has no accounts, external providers ship **disabled**, and
signups default **off** — so a freshly deployed instance has no way in.
The script builds `fly/internal.db` locally with
`scripts/bootstrap_internal_db.py`, which:

- creates the first admin (`--username`, default `admin`; generates and
  prints a password unless `--password` is given; idempotent — skips if an
  enabled admin exists);
- enables discord/github/bluesky and sets their signups policy
  (`--signups off|approval|auto`, default `approval` = new identities queue
  for admin approval).

It calls the same `db.*` functions as the `datasette accounts` CLI, but
constructs `Datasette(plugins_dir="plugins")` itself — the CLI's
`enable-provider` validates keys against the provider registry, and it has no
way to load `--plugins-dir` modules. Startup hooks also create the bluesky
sample's flow table.

Then it uploads via `fly ssh sftp` (put to `/mnt/internal.db.new`, swap into
place clearing stale `-wal`/`-shm`, since sftp won't overwrite) and restarts
the app. **The upload replaces the remote internal DB wholesale** — it's a
first-boot tool. Afterwards manage everything signed in as the admin, or on
the machine:

```sh
fly ssh console -C "sh -lc 'cd /app && uv run datasette accounts list -i /mnt/internal.db'"
```

(That `fly ssh console` route is also the no-file-copy alternative for later
account fixes — e.g. `accounts reset-password`, `accounts enable-provider`.)

## Redeploying

- Code change in the checkout (plugin or samples): `scripts/deploy.sh` — it
  re-syncs everything.
- Changed `.env`: `scripts/secrets.sh` then `scripts/deploy.sh` (secrets are
  staged, applied on deploy; or drop `--stage` in the script to apply
  immediately with a restart).
- Bumped the plugin version: update the wheel filename in
  `pyproject.toml`'s `[tool.uv.sources]` first (sync.sh errors loudly if
  they disagree).

## Permissions: --default-deny + ACL

`run.sh` starts Datasette with `--default-deny` (core's default view-* allow
rules are skipped, so databases/tables are invisible to everyone) plus
`-s permissions.view-instance true` (allow block `true` matches anonymous,
so the homepage and sign-in surface stay reachable). Visibility is then
granted per resource through **datasette-acl** (in the deps; accounts admins
get its admin UI via the plugin's `grant_acl_admin` bridge, on by default):

- groups: `/-/acl/groups`
- a database (`view-database`): `/-/acl/resource/database/congress-legislators`
- a table (`view-table` — per table; grant to a group to scale):
  `/-/acl/resource/table/congress-legislators/<table>`

A user needs `view-database` on the database AND `view-table` on a table to
read that table; grants take effect on the next request. The raw JSON API
(`/-/acl/api/resource/.../grant`) takes the account's **ULID id**, not the
username — the HTML UI resolves people properly, so prefer it.

## Notes

- **One machine, no HA** (`--ha=false` in deploy.sh): the volume attaches to
  a single machine; two machines would get separate, divergent DBs.
- `DATASETTE_SECRET` must stay stable (signed cookies + OAuth state);
  secrets.sh generates it once and writes it back to `.env`.
- Local smoke test of exactly what the container runs:

  ```sh
  DATASETTE_SECRET=dev DATASETTE_BLUESKY_PUBLIC_URL=http://localhost:8080 \
    uv run datasette --internal internal.db data.db --create \
    --plugins-dir plugins -h 0.0.0.0 -p 8080
  ```

- flyctl only honors `DOCKER_HOST` (not Docker contexts); deploy.sh exports
  OrbStack's socket automatically when present.
