datasette-auth-basic-login

Database-backed login for Datasette, with admin account management and a Svelte/Vite/TS frontend. Pre-build design spec.

Awaiting Fable 5 review — no code written yet

This is the single-page render of the plan in plans/start/*.md. Every decision was made deliberately in a grilling session; alternatives and rationale are in the decisions log below and in plans/start/09-decisions-log.md.

What it is

A Datasette auth plugin modeled on datasette-auth-passwords, but with three departures the user asked for:

  • Accounts in the internal database — created/managed at runtime, not hard-coded in plugin config.
  • An admin permission — a Datasette 1.0 action that provisions accounts, revokes access, resets passwords, and grants admin.
  • A Svelte 5 / Vite / TypeScript frontend — built per datasette-alex-fullstack-skill (login, admin dashboard, self-service account page).

Scope boundary: this plugin owns identity (accounts, passwords, sessions, one is_admin flag). It does not own resource-level authorization — that is delegated to datasette-acl / config allow blocks, which consume the actor we emit:

{ "id": "<ULID>", "username": "alice", "is_admin": true }

id is an immutable ULID and the permanent join key to user-profiles and acl; username is the mutable login credential.

Architecture at a glance

Targets

Datasette ≥ 1.0a23 · Python ≥ 3.10 · 1.0 action API (register_actions / datasette.allowed)

Backend

datasette-plugin-router · Pydantic · sqlite-migrate · python-ulid

Frontend

Svelte 5 runes · Vite 7 · datasette-vite · type-safe pipelines (Pydantic→TS, OpenAPI→TS)

Hashing

PBKDF2-HMAC-SHA256 (480k), copied verbatim from auth-passwords · stdlib, no new deps

Data model — internal DB

Migration namespace datasette-auth-basic-login.internal. No collision with user-profiles' tables.

Authentication flow

Sessions

Server-side, revocable. token = secrets.token_urlsafe(32); store sha256(token); cookie carries sign(token); absolute 14-day expiry.

Cookie

ds_auth_basic_login_session (own, not ds_actor) · httponly · SameSite=Lax · Secure on HTTPS.

actor_from_request re-checks the session + account on every request — so disabling an account or deleting a session logs the user out on the next request, and renames / admin-flag changes take effect live.

cookie → unsign → sha256 → sessions lookup → not expired?
      → users lookup → not disabled? → {id, username, is_admin}
  • Login: Svelte /-/login → JSON POST → verify → mint session → set cookie → redirect.
  • Logout: own POST /-/logout → DELETE the session row + clear cookie (core's /-/logout only clears ds_actor).

Admin & permissions

  • Register action datasette-auth-basic-login-admin, self-answered as root OR users.is_admin=1 (via permission_resources_sql) — composable with acl/config, visible to introspection.
  • Bootstrap via datasette --root: root is always admin, creates the first real admin, then steps back. No secrets in config.
  • Admin UI (Svelte, /-/admin/users): create · reset-password · toggle-admin · disable (reversible, kills sessions) · hard-delete · unlock · list/revoke sessions · log-out-everywhere.
  • Last-admin guard: refuses to disable/delete/demote the final admin (synchronous count-then-write, HTTP 409), like acl's last-manager guard.

Self-service & brute force

Self-service

Users change their own password at /-/account. Admin-created accounts & resets set must_change_password, forced at next login (enforced server-side too). Changing password revokes other sessions.

Brute force — hard lockout

5 failures → lock 15 min (both configurable). Auto-unlock on timeout; admin manual unlock. All attempts audited.

Accepted tradeoff (D15): account-keyed lockout enables a login-DoS — an attacker who knows a username can lock that real user out. Mitigations (IP-keying, soft throttle) are future and schema-compatible.

Compatibility with datasette-user-profiles

  • Emit a stable string id = profiles' actor_id join key.
  • Seed the directory via the datasette_user_profile_seeds hook (fill-missing, idempotent); never write profiles' tables directly.
  • profiles owns display_name/email/bio/avatar as the editable directory copy.
  • Accepted consequence: runtime-created accounts appear in the directory only after a restart or the user's first self-edit (D13, Option 2).
  • Plugin works with or without user-profiles installed (guarded import).
